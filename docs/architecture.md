# 大学生模拟器 v0.2 架构边界

## 设计目标

当前版本优先保证三件事：

1. 核心规则必须全部由代码决定。
2. AI 只能做表达层，不能做判定层。
3. 模块边界要足够清晰，便于继续扩展玩法。

## 分层约束

### 数据层

目录：

- `lib/supabase`
- `db`

职责：

- 管理 Supabase client 与 repository。
- 管理 `runs`、`monthly_states`、`game_event_logs`、`ai_reports`、`resume_items`。
- 持久化 run 当前状态、月度快照、关键日志、履历条目和 AI 输入输出。

禁止：

- 在 repository 里写游戏规则。
- 在数据层里拼玩家可见叙事。
- 依赖 AI 返回值来反向修正规则结果。

### 规则层

目录：

- `core/game-engine`
- `core/generators`
- `core/resolvers`
- `data`

职责：

- 随机生成开局属性。
- 维护周历推进、行动即时结算、课程策略后果、月度事件、学期结算和毕业风险。
- 产出结构化事实摘要，供 UI 和 AI 表达层消费。

禁止：

- 把关键规则放进 prompt。
- 依赖页面组件状态做结算。
- 让 AI 决定数值、结果或事件是否发生。

### UI 层

目录：

- `app`
- `components`

职责：

- 展示开局信息、当前状态、周历、即时结果卡、行动入口、月结算、月记、履历和结局页。
- 通过 server actions 调用整合层。
- 把规则层结果组织成可读界面。

禁止：

- 在页面里硬写核心规则。
- 在组件里私自重算状态。
- 用 prompt 替代规则层判断。

### AI 表达层

目录：

- `lib/ai`
- `core/prompts`

职责：

- 只生成两类内容：
  - 月记
  - 结局报告
- 只消费规则层给出的结构化摘要。
- 保留 AI 输入摘要和最终输出，便于排查。

禁止：

- AI 参与课程、行动、事件、学期、毕业判定。
- AI 编造未发生的重要事实。
- AI 在 prompt 中偷偷承载规则逻辑。

### 整合层

目录：

- `lib/demo`
- `app/actions.ts`

职责：

- 串联规则层、数据层和 AI 表达层。
- 负责 create run、advance turn、保存月度快照、写日志、写 AI 报告。
- 保证页面只需要调用整合入口，不需要跨层手工拼流程。

## v0.2 新的玩法流

当前主流程如下：

1. 开局页创建 run。
2. 主游戏页按周显示当前月的 4 周周历。
3. 玩家每次只提交 1 个行动。
4. 规则层立即结算本次行动，并即时更新状态。
5. 普通行动会消耗当前周时间池；只有时间耗尽或玩家主动结束时才进入下一周。
6. `big_meal`、`ask_family`、`skip_class` 这类 0 时间动作不推进周历，但会立刻改动状态或释放本周可用时间。
7. 主游戏页会把最近一次行动渲染成即时结果卡，告诉玩家这一轮变了什么、这周还剩多少时间、下一步还能做什么。
8. 第 4 周真正结束后，规则层汇总月度快照、随机事件、前台叙事事实和 AI 月记输入。
9. 整合层保存 `monthly_states`、`game_event_logs`、`ai_reports`、`resume_items`。

## 日志拆分

### 后台日志

来源：

- `game_event_logs`

用途：

- 保留系统留档和排查信息。
- 记录 run 创建、单次 turn 结算、月结算、事件触发等。

### 前台日志

来源：

- 规则层月度结构化摘要
- `monthly_states.snapshot_json`

用途：

- 用中文、玩家视角回顾“这个月怎么过的”。
- 把机器事实翻译成可读叙事，但不改写事实本身。

## AI 边界

AI 永远在规则之后：

1. 规则层先算出事实。
2. 整合层把结构化摘要传给 AI。
3. AI 只负责把这些事实写成月记或结局报告。

这意味着：

- AI 不能决定奖励、惩罚、通过与否。
- AI 不能新增规则层没有给出的关键事件。
- fallback 文案也必须遵守同样边界。

## 当前已知限制

- 现在的“按次行动”已经支持周内多动作，但还没有细到每天时间块。
- 随机事件池已做成数据驱动，但体量仍然偏小。
- 前台日志已经和后台日志分离，但更细的角色关系叙事还没有展开。
- 目前仍然是本地 Demo 优先，不包含登录和复杂权限模型。
