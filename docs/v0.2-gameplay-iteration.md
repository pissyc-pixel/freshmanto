# 大学生模拟器 v0.2 玩法迭代边界说明

## 本轮目标

本轮不是继续“加功能”，而是优先把 v0 的工程闭环修成更像可玩的生活模拟 Demo。

必须坚持：

- 核心规则继续全部由代码判定
- AI 只负责月记和结局报告
- 页面负责展示和交互，不承担核心结算逻辑
- 各模块边界继续保持清晰，便于后续扩展

## 当前试玩问题与代码根因

### 1. 行动仍是整月批处理

当前问题：

- 玩家一次性选完整个月行动，无法根据上一步结果调整
- 行动体验更像填表，不像“过日子”

根因位置：

- `types/game.ts` 仍以 `MonthlyActionPlan` 为中心
- `core/game-engine/monthly.ts` 使用 `resolveMonthlyTurn` 一次结算整月
- `lib/demo/run-service.ts` 以 `advanceDemoMonth` 为主入口
- `app/actions.ts` 和 `components/action-plan-form.tsx` 也是一次性提交 3 个槽位

### 2. 时间结构不直观

当前问题：

- 30 天自由/半自由/忙碌的表达对玩家不够直观
- 玩家无法感知“周内忙、周末空”的节奏

根因位置：

- `core/resolvers/schedule.ts` 仍是 30 天模式
- `app/game/page.tsx` 只是把日程表平铺展示，没有形成周历心智

### 3. 日志只有后台视角

当前问题：

- 现有日志更像系统/开发记录，不像玩家在回看“这个月经历了什么”
- 前台缺少中文、具体、可读的过程描述

根因位置：

- `lib/demo/run-service.ts` 的 `createRunLog` / `createMonthlyLogs` 只产出后台日志
- `components/log-feed.tsx` 直接展示系统日志
- `app/settlement/page.tsx` / `app/resume/page.tsx` 没有区分前台日志与后台日志

### 4. 月记仍偏结算摘要

当前问题：

- 月记和 fallback 文案偏结算播报
- 缺少角色自述、情绪感和生活感

根因位置：

- `lib/ai/reports.ts` fallback 写法仍是总结语气
- `core/prompts/monthly-journal.ts` 的约束虽正确，但输出风格还不够拟人

### 5. 经济、压力、学习平衡不足

当前问题：

- 没有固定支出，生活成本压力感不够
- 学习收益偏高，容易变成无脑最优解
- 压力高几乎没有明显惩罚
- 缺少“吃大餐”等生活消费动作

根因位置：

- `core/resolvers/actions.ts` 的收益设计偏线性，学习收益过高
- `core/game-engine/monthly.ts` 只做月度 allowance 叠加，没有固定月支出
- `core/resolvers/events.ts` 和 `data/events.ts` 的状态事件数量与触发维度过少

### 6. 翘课代价链路过扁平

当前问题：

- 当前主要通过课程风险和简单数值变化体现
- 没有形成“点名风险 / 平时分风险 / 代签花钱 / 补救成本”的后果链

根因位置：

- `core/resolvers/attendance.ts`
- `core/game-engine/monthly.ts`

### 7. 前台文案存在乱码

当前问题：

- README、部分页面、组件、事件文本和 fallback 文案里存在编码乱码
- 直接破坏试玩体验，也会影响后续叙事重构

重点文件：

- `README.md`
- `docs/architecture.md`
- `app/game/page.tsx`
- `app/settlement/page.tsx`
- `app/journal/page.tsx`
- `components/log-feed.tsx`
- `components/action-plan-form.tsx`
- `lib/demo/options.ts`
- `data/events.ts`
- `core/resolvers/actions.ts`
- `lib/ai/reports.ts`

## 本轮 3 个 subagent 的职责边界

为尽量减少冲突，本轮不按“粗目录”而按“精确文件所有权”并行。

### Subagent A：时间流与行动系统重构

目标：

- 把“整月批处理”重构为“按次行动 -> 即时推进 -> 即时结算”
- 把时间表达改成周历
- 修正主流程，让玩家可以一步一步选择
- 重构翘课后果链路，但不负责经济平衡细节和叙事润色

优先拥有文件：

- `types/game.ts`
- `core/game-engine/monthly.ts`
- `core/game-engine/index.ts`
- `core/resolvers/schedule.ts`
- `core/resolvers/attendance.ts`
- `lib/demo/run-service.ts`
- `lib/demo/server.ts`
- `app/actions.ts`
- `app/game/page.tsx`
- `components/action-plan-form.tsx`
- `components/time-block-strip.tsx`
- `tests/demo-run-service.test.ts`

尽量不要改：

- `core/resolvers/actions.ts`
- `core/resolvers/events.ts`
- `data/events.ts`
- `lib/ai/*`
- `core/prompts/*`
- `app/settlement/page.tsx`
- `app/journal/page.tsx`
- `components/log-feed.tsx`

### Subagent B：经济 / 状态平衡 / 随机事件

目标：

- 加固定支出和“吃大餐”
- 削弱学习收益并加入递减与状态惩罚
- 给高压力增加行动负反馈
- 扩展最小随机事件系统和状态触发事件

优先拥有文件：

- `core/resolvers/actions.ts`
- `core/resolvers/events.ts`
- `core/resolvers/semester.ts`
- `data/events.ts`
- `tests/core-rules.test.ts`

可新增但尽量局部：

- `core/resolvers/*` 下新的平衡辅助文件
- `data/*` 下新的事件表或静态表

尽量不要改：

- `app/**`
- `lib/ai/**`
- `core/prompts/**`
- `lib/demo/run-service.ts`
- `lib/demo/server.ts`

说明：

- 如果需要新增状态事件相关类型，优先让主 agent 在整合阶段统一落到 `types/game.ts`，避免与 Subagent A 同时改同一类型文件。

### Subagent C：前台日志与拟人月记重构

目标：

- 把日志拆成前台日志和后台日志
- 把月记改成更像大学生日记的自述
- 修复前台文案乱码
- 优化月结算/日志/月记页的可读性

优先拥有文件：

- `app/settlement/page.tsx`
- `app/journal/page.tsx`
- `app/resume/page.tsx`
- `app/ending/page.tsx`
- `components/log-feed.tsx`
- `components/report-preview.tsx`
- `components/history-timeline.tsx`
- `components/fact-list.tsx`
- `components/nav-bar.tsx`
- `lib/ai/reports.ts`
- `core/prompts/monthly-journal.ts`
- `core/prompts/ending-report.ts`
- `lib/demo/options.ts`

可按需修复乱码的玩家可见文本文件：

- `README.md`
- `docs/architecture.md`

尽量不要改：

- `core/game-engine/**`
- `core/resolvers/actions.ts`
- `core/resolvers/events.ts`
- `lib/demo/run-service.ts`
- `app/actions.ts`

## 主 agent 保留的整合职责

以下文件默认由主 agent 在阶段 3 统一收口，避免 subagent 间来回争用：

- `README.md` 最终范围与运行说明
- `docs/architecture.md` 最终边界更新
- `docs/v0.2-gameplay-iteration.md`
- 任何跨模块接口适配
- 任何需要同时理解 A/B/C 三块结果的类型与测试修复

## 并行推进策略

1. 先让 Subagent A 改“时间流和玩法骨架”
2. 同时让 Subagent B 改“收益曲线、经济成本、随机事件”
3. 同时让 Subagent C 改“前台叙事、日志和月记表达”
4. 最后由主 agent 统一接线：
   - 对齐类型
   - 修复接口不匹配
   - 补 integration tests
   - 更新 README / docs

## 阶段 3 的联调重点

主 agent 联调时重点确认：

- 一次行动就会推进一个明确时间格，并立即结算
- 周历表达足够直观
- 前台日志与后台日志分离
- 月记是拟人自述，不是系统播报
- 固定支出与“吃大餐”都能真实影响取舍
- 学习收益不再无脑最优
- 高压力时有明显负反馈
- 至少有若干状态驱动随机事件能真实触发

## 已知风险

- `types/game.ts` 同时承载日程、行动、总结、风险等多类结构，整合时最容易冲突
- `lib/demo/run-service.ts` 目前仍是月度入口，本轮大概率需要较大改造
- 前台乱码分布较广，必须避免在多个 subagent 中重复修同一批文本
